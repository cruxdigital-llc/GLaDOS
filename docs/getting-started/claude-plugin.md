# Claude Code (plugin install)

For teams who run Claude Code — Anthropic's terminal coding agent — and want
one shared GLaDOS install distributed as a Claude Code **plugin**, instead of
compiling commands into every repository separately.

If you only care about one repo, the simpler script install in
[claude-code.md](claude-code.md) is usually the better fit. Not sure which
runtime you have? See the [chooser](README.md).

## How plugin mode differs

In every other mode, the workflows are compiled against *your project's*
`glados.yaml`. A plugin is shared across many projects, so its artifacts are
compiled from the neutral template `glados.yaml.example` instead, and every
workflow starts with a **bootstrap guard**: a compiled preamble telling the
agent to read the current repo's `glados.yaml` at run time and let it govern
wherever it differs — and, if the repo has none, to stop and create one
(via `/glados:init`) before doing consequential work.

## Prerequisites

- Claude Code installed (`claude` on your PATH).
- A checkout (or your team's fork) of the GLaDOS repository — the repository
  itself is the plugin: it carries `.claude-plugin/plugin.json` plus the
  pre-compiled artifacts.

## Install (consumer)

Load the plugin from a local checkout for the current session:

```bash
claude --plugin-dir /path/to/GLaDOS
```

Or, if your team publishes GLaDOS through a Claude Code plugin marketplace
(a catalog repo with a `.claude-plugin/marketplace.json`):

```bash
claude plugin marketplace add your-org/your-marketplace-repo
claude plugin install glados@your-marketplace
```

`claude plugin validate /path/to/GLaDOS` confirms the manifest is well-formed.

## What the plugin contains

These artifacts are checked into the GLaDOS repository and regenerated by the
maintainer command below — consumers never run the compiler:

```
GLaDOS/  (the plugin)
├── .claude-plugin/plugin.json       # plugin manifest (name: glados)
├── skills/<name>/SKILL.md           # 25 entry points: 15 workflows + 10
│                                    #   renamed-v1-name shims (+ init)
├── compiled/claude-plugin/*.md      # the 15 compiled workflow bodies the
│                                    #   skills point at via ${CLAUDE_PLUGIN_ROOT}
└── glados.yaml.example              # template /glados:init copies into repos
```

## What still lives in your workspace

The plugin brings the *workflows*; each consuming repo still owns its state:

- **`glados.yaml`** — created per repo. Run `/glados:init` (ships with the
  plugin) to copy the template in, set the required `phase:`, and scaffold
  `product-knowledge/`.
- **`.glados/runs/`** — run records are committed to the repo the run
  happened in, never to the plugin.
- **`product-knowledge/`** — the project's own standards, personas, and
  status file.

## First run

In Claude Code, in a repo of your choice:

```
/glados:init
```

then a read-only first workflow:

```
/glados:review-codebase
```

Plugin skills use the same `/glados:<name>` syntax as the script install.

## Verify

The plugin repo itself carries a vendored checker; from the GLaDOS checkout:

```bash
python .glados/glados.py check --target .
```

Expected: `glados check: OK — no drift, checks pass, manifest hash current`.
In consuming repos there is nothing compiled to check — the bootstrap guard
plus each repo's `glados.yaml` do the work at run time.

## Guard hook

Claude Code's Stop hook (blocks a session ending mid-run without a committed
run record) is wired per consuming repo in `.claude/settings.json` — snippet
in [hooks/README.md](../../hooks/README.md). The snippet expects the guard
script at `.glados/hooks/claude-stop-hook.py`, which plugin-only repos don't
have; copy `hooks/claude-stop-hook.py` from the GLaDOS checkout to that path
(or point the snippet wherever you keep it).

## Maintainer corner: regenerating the artifacts

After changing workflow sources or bumping the GLaDOS version, regenerate the
checked-in plugin artifacts from the repo root:

```bash
python bin/glados.py compile-plugin
```

(shorthand for `python bin/glados.py install --mode claude-plugin --target .`).
It rewrites `compiled/claude-plugin/` and the generated `skills/` stubs,
pruning stale generated stubs while leaving hand-written skills — like
`init` — untouched.

## Gotcha: phase-neutral by design

The plugin's compiled text deliberately contains no project `phase:` — it was
built from the example template, and the one judgment-call sentence stamped
into every workflow is the cautious baseline ("optimize for doing no harm").
Your repo's real posture comes entirely from its own `glados.yaml` at run
time. If runs feel oddly conservative, the repo is probably missing its
`glados.yaml` — the bootstrap guard is doing exactly what it says.
